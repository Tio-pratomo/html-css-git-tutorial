## Materi: Pengetahuan & Konsep

Sesi 20 ini adalah jembatan: dari **UI Bulma** ke dunia **backend & API**.

Tujuannya supaya kamu paham dulu “gambar besarnya” sebelum nanti kita pilih backend konkret (misalnya Node/Express, Go, atau lainnya).

Gambaran arsitektur web modern yang relevan buat kamu sekarang biasanya dibagi 3 lapis:

1. **Frontend (client)**
   - Semua yang sudah kamu bangun dengan Bulma: halaman HTML, CSS, JavaScript di browser.
   - Tugasnya: menampilkan UI, menangkap input user (form, klik), dan mengirim/ambil data lewat HTTP (biasanya JSON) ke backend.

2. **Backend / API server**
   - Program yang jalan di server (bisa Node.js, Go, PHP, dll.).
   - Menyediakan **REST API**: URL tertentu yang bisa dipanggil frontend pakai HTTP method (`GET`, `POST`, `PUT`, `DELETE`, dll.), biasanya kirim/terima data dalam bentuk JSON.
   - Contoh sederhana:
     - `GET /api/books` → ambil daftar buku.
     - `POST /api/books` → tambah buku baru (data dikirim dalam body JSON).

3. **Database**
   - Penyimpan data permanen (MySQL, PostgreSQL, MongoDB, dsb.).
   - Backend yang bicara ke database (frontend **tidak langsung** akses DB).

Alur sederhananya ketika user pakai dashboard Bulma kamu:

1. User buka `inventory.html` di browser.
2. JavaScript di halaman itu memanggil `GET /api/books` ke backend (via `fetch`).
3. Backend baca data dari database, balas JSON, misalnya:

   ```json
   [
     { "id": 1, "title": "Mastering Bulma", "price": 150000 },
     { "id": 2, "title": "React for Beginners", "price": 175000 }
   ]
   ```

4. JS di frontend menerima JSON ini dan **mengisi tabel** di halaman Bulma.
5. Saat user submit form “Add Book”, JS kirim `POST /api/books` dengan body JSON berisi data buku; backend simpan ke DB, lalu balas status (sukses/gagal) dan mungkin data buku yang baru dibuat.

Konsep ini mengikuti style **REST API**: resource diwakili oleh URL, diakses pakai HTTP methods, dan data diformat sebagai JSON.

## Praktik

Di sesi ini kita belum tulis kode backend (konseptual dulu), tapi kamu bisa mulai membayangkan:

- Halaman mana di proyek Bulma kamu yang idealnya **diisi oleh API**:
  - `inventory.html` → data tabel buku.
  - `quick-edit.html` → update stock & price via API.
  - `dashboard.html` → angka statistik di card (total buku, total penjualan, dsb.).

- Untuk tiap halaman itu, bayangkan endpoint API‑nya:
  - List buku: `GET /api/books`.
  - Detail buku: `GET /api/books/:id`.
  - Tambah buku: `POST /api/books`.
  - Update stok/harga: `PUT /api/books/:id` atau `PATCH /api/books/:id`.
  - Hapus buku: `DELETE /api/books/:id`.
